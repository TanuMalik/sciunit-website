I. Overview

Sciunit is a cloud-based framework for enabling reproducible computational analysis. Currently, the framework is used within the geosciences domain to automatically create Docker containers of geoscience applications. The framework consists of two components: the client (i.e., Sciunit-CLI) and the server (i.e., the Sciunit hub). We present in detail the Sciunit-CLI in this document.

II. Getting Started

This section provides information to new users how to get started using the Sciunit-CLI on Linux machines (all of the parts presented in this document were tested on Ubuntu 16.04). In particular, we show how to install and how to configure Sciunit-CLI in the installation and configuration sections. Next, we present some running examples with Sciunit-CLI in the tutorial section. Finally, we report some common issues may occur during running time and our recommended solutions for those issues.

IV. Configuration

Follow these three easy steps prior to running Sciunit-CLI.

  1. Globus and Hydroshare accounts

    We request users to acquire a Globus and a Hydroshare account:

    • a Globus account from www.globus.org

    • Obtain a Hydroshare account from www.hydroshare.org


    Since in our Sciunit-CLI, we support a secure and efficient data transfer protocol using Globus and a trusted and reliable publish environment with Hydroshare, a Globus account is required to share and publish sciunit packages at Sciunit.org; whereas a Hydroshare account is used to publish sciunit packages at Hydroshare.

    However, Sciunit.org is currently not set up for hosting sciunits from multiple users. Therefore, once obtaining a Globus account, Globus username should be emailed to scidataspace@gmail.com in order to share and publish sciunit packages at Sciunit.org.

  2. Configure Sciunit-CLI

    The configuration of globus account will be launched automatically in the first time running Sciunit-CLI. This information is recorded in ~/.sciunit folder in order to avoid asking for this information in the later times. In this step, Sciunit-CLI also sets up your machine as an endpoint for uploading sciunits to the server. Here is the details of this first time running Sciunit-CLI.

    After installation of Sciunit-CLI, users can launch Sciunit-CLI by using $ sciunit-cli.py. Here is the output:

    $ sciunit-cli.py
    Enter Globus username: <Please enter your Globus username>
    Globus Password:
    Globus local-endpoint name: <Please provide a name for this machine>
    GC Setup Key: <32 bit key from Globus>
    Endpoint created successfully
    Configuration directory: /home/ubuntu/.globusonline/lta
    Contacting relay:globusonline.org:2223
    Done!
    Enter "stop" to end sesssion
    Commands start with --
    undefined >

    In later times, no Globus configuration information will be asked. The interface is shown in the fowlling figure:


    The configuration of Hydroshare account will be done in the first time users publish their packages. Similar to Globus configuration, this configuration process only happens in the first time publishing a sciunit package to Hydroshare. Users' Hydroshare authentication information is also stored in ~/.sciunit folder. For more details about Hydroshare configuration, see the Example 4: "Publish a sciunit package to hydroshare" in the Tutorial section.

  3. Run "helloworld" with Sciunit-CLI

    Example Sciunit applications can be found under the examples directory. Applications are included with a README file to help run them.

    1. Run Sciunit-CLI from command-line: $ sciunit-cli.py

      Note that, in this example we used $ sudo sciunit-cli.py to run in root mode becuase some applications (e.g. hellowordl.sh) may require it due to the permission conflicts between users. Without these conflicts, this root mode is not necessary.

    2. Sciunit-CLI allows users to manage many application sor projects. Each of them can be contained in a sciunit composed by many packages. to start a sciunit, users can use command: --sciunit start <sciunit_name>

    3. Package the HelloWorld app in Examples directory, with command --package </PATH/TO/execution_file>

      This generates package_hashID (10d47b6ca8e4d26d37bcb7fd29b826c8fa951a29). This is the ID of this package. The package_id is unique value in within a sciunit.

    4. List all packages in this sciunit: --package list

    5. Re-run the package in this sciunit with its id: --repeat <pacakge_id>

    6. To exit Sciunit-CLI: --stop

V. Sciunit Tutorial

This hands-on tutorial give a quick ~20 minute taste of creating, sharing, and publishing sciunits using the CLI. The created sciunits are geoscience domain specific and are published to Hydroshare, a geoscience publish repository

  1. Introduction: What are sciunit?

    Sciunits are efficient, lightweight, composed by self-contained packages of computational experiments that can be guaranteed to repeat or reproduce regardless of deployment issues. Scientists build sciunits for self-curation of experiments, for sharing their experiments with others, and for publishing their experiments on community hubs. Sciunits aim to eliminate "does not work on my computer" problems in the scientific community.

    The Sciunit-CLI helps you build sciunits. Packaging of application can be accomplished with a single command, using a statement like:

    --package </PATH/TO/execution_file>

    This command will build a lightweight, self-contained package, independent of any programming language or specific OS. This package can be described by three kinds of metadata: annotation metadata, provenance metadata, and version metadata. Provenance and version metadata can be automatically generated. The package along with its metadata can be, preserved, shared, or published.

  2. Example 1: Package a single science application

    This first example creates a sciunit from the Food inspection evaluation application (FIE) (this is an open-source project created by the City of Chicago and available for downloading at: Github.com/Chicago/food-inspections-evaluation) which was written in R and reads data from the food_inspections.Rds in the 'default' directory and outputs a violation_dat.Rds file.

    1. Start a Sciunit-CLI: $ sciunit-cli.py

    2. Start a sciunit: --sciunit start <sciunit_name>

    3. Run and create a package in the current sciunit: --package </PATH/TO/execution_file>

    4. In this example, we careted a sciunit, called Mysciunit, and run the execution file (runbashRandomForestOneStep.sh). This .sh script file will execute an R program in FIE.

    5. A 32-bit package_hash ID is created and associates with the current package. This package_id is unique within the current sciunit (Mysciunit).

    6. Created package will be inserted to our database (i.e., vvpkg.bin and vvpkg.db) in ~/.sciunit/package directory.

    7. Exit Sciunit-CLI: --stop

  3. Example 2: Package and repeat several applications

    9. Repeat another package: --repeat <package_id>

VI. Troubleshooting

Known Errors:

  1. Problem: Setup script exited with error in docker-py setup command: Invalid environment marker: python_version <"3.5"

    Solution: $ pip install setuptools --upgrade

  2. Problem: ssl.SSLError: [SSL: CERTIFICATE_VERIFY_FAILED] certificate verify failed (_ssl.c:590)

    Solution: $ export PYTHONHTTPSVERIFY=0

  3. Problem: AttributeError: 'Hydroshare' object has not attribute 'resource'

    Solution: update ehe hs_restclient libraries by using: $ sudo pip install -U pip, $ sudo pip install -U hs_restclient
